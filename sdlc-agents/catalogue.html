Cross-cutting techniques & the program behind them

The reusable AI practices that repeat across the delivery lifecycle — not tied to any one product — and how the practice itself is run, tooled, and governed.

Public-safe summary of the internal "AI in SDLC" space. Individual and per-ticket data intentionally omitted.

01

Cross-cutting techniques

The same handful of techniques recur everywhere. Each is a repeatable AI workflow a team can apply on any codebase.

Test generation & coverage

Turn requirements and existing code into tests, and raise coverage on modules that lack it.

  • Unit tests — Angular/Jest, PHPUnit, Flutter widget tests
  • Manual test-case creation from acceptance criteria
  • Coverage enhancement on existing modules
  • Test-automation suites for regression

Browser & E2E automation

Drive real browser flows through the Playwright MCP server.

  • Stand up a shared Playwright MCP server
  • Automate flows — login, password validation, form journeys
  • Modernize — convert legacy Playwright JavaScript to TypeScript

Accessibility remediation

WCAG 2.1 AA fixes validated against the live DOM with axe-core.

  • Fix contrast, semantic tables, labels, keyboard/focus, alt text
  • Multi-page content (e.g. EHR charts) and contrast-tool verification
  • VPAT documentation for compliance

Observability & reliability

Read and act on production signals through the Datadog MCP.

  • Log & APM analysis, error triage
  • Predictive anomaly detection & resource-allocation optimization
  • SLOs & monitors for services
  • DORA / engineering-metrics reports auto-generated by a metrics agent

Upgrades & migrations

Framework, runtime, and language moves done with AI assistance.

  • Frameworks — Angular 19→20; Flutter & Riverpod
  • Runtime — PHP 8.4 + Laravel 12
  • Ports — MVC/Razor → Angular; JavaScript → TypeScript
  • Cleanup — dependency & obsolete-code removal

Code quality & review

Keep changes clean and aligned to team standards.

  • Optimization — memory & timeout issues
  • Refactoring for readability & maintainability
  • AI code review via the Atlassian MCP, against Jira requirements

Feature delivery (prompt-first)

Build features from a prompt-first workflow with approval gates.

  • Conditions-based feature implementation
  • Mobile — biometric auth, deep linking, FCM push notifications
  • Localization of existing pages

UI from design

Turn design artifacts into working code.

  • Figma & screenshots → Angular components
  • Shadow-root UI element access in JavaScript

DevOps & configuration

The glue around delivery — environments, secrets, and hardening.

  • Git remote-branch pointing changes
  • Secrets — retrieve from GitHub Actions
  • Environments — lower-env DB pointing (Android/iOS)
  • Hardening — SSL cert issues, JScrambler
02

Program & foundation

How the practice is run — the people, tooling, and guardrails that make the techniques above repeatable and safe.

AI Champions

A champions network that drives adoption across teams.

  • Share patterns and proven prompts
  • Onboard teams onto the toolchain
  • Curate new use cases into the shared space

Tooling

The assistants and connectors the work runs on.

  • GitHub Copilot — chat + CLI, per-repo agents & skills in .github/
  • Rovo — including "Rovo in QE" and a shared prompt repository
  • MCP servers — Playwright, Datadog, Atlassian

Custom agent toolkits

Reusable agent + skill bundles dropped into a repo or ~/.copilot/.

  • MedHub suite — ambs-debug, medm-analysis, medm-implement, review-mr, datadogagent
  • Architecture & Planning — AscArchDocAgent
  • Metrics — a DORA metrics agent
  • Shared skills — one skill system reused across agents

Shared knowledge

So no one re-explains context from scratch.

  • Prompt repository & a use-case template
  • Confluence knowledge exports loaded into agents
  • Per-project docs — architecture, database, features

Onboarding

Get a new engineer productive on the toolchain fast.

  • New-member guide — tools, access & usage
  • Setup scripts for agents, skills & credentials

Guardrails & governance

The same safety model across every agent.

  • Preview before apply — diff shown, nothing written until approved
  • No auto-push / no auto-merge — reviewers stay responsible
  • SELECT-only SQL, duplicate checks, worklog confirmation
  • Metrics — DORA / engineering reports auto-generated from Jira
